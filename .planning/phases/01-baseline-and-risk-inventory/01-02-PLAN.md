# Plan 02: Build Concern Trace Matrix

<objective>
Convert high-priority mapped concerns and test gaps into a traceable matrix with requirement IDs, evidence paths, subsystem grouping, severity, v0.3 dependency, and disposition.
</objective>

<threat_model>
This plan reads planning documents and writes a markdown inventory. It does not modify source or execute product code. Risks are misclassifying concerns, losing source evidence, and accidentally expanding Phase 1 into feature planning.

Mitigations:
- Use `.planning/codebase/CONCERNS.md` as the primary evidence source.
- Include source evidence paths for every matrix row.
- Use only the approved disposition labels: `Must fix`, `Should fix`, `Can defer`, `Out of scope`.
- Mark new capabilities and major rewrites as deferred unless they directly block Phase 1-6 requirements.
</threat_model>

<must_haves>
<truths>
- D-05: Rank only high-priority concerns and test gaps.
- D-06: Rank by risk first, then v0.3 dependency as tie-breaker.
- D-07: Group by subsystem and include severity labels.
- D-08: Defer new capabilities and major rewrites unless they directly block Phase 1-6 requirements.
- D-09: Produce both a requirements-to-concerns matrix and a prioritized task list.
- D-10: The trace matrix must include concern, evidence path, requirement ID, subsystem, severity, v0.3 dependency, and recommended disposition.
- D-11: Use disposition labels `Must fix`, `Should fix`, `Can defer`, and `Out of scope`.
- D-12: Store the traceability work as a dedicated markdown artifact under `.planning/phases/01-baseline-and-risk-inventory/` and summarize it later.
</truths>
<outputs>
- `.planning/phases/01-baseline-and-risk-inventory/01-INVENTORY.md` exists.
- The inventory contains a concern trace matrix and prioritized task list.
</outputs>
</must_haves>

<action>
Create `.planning/phases/01-baseline-and-risk-inventory/01-INVENTORY.md` with this top-level structure:

```markdown
# Phase 1 Baseline and Risk Inventory

## Scope
## Concern Trace Matrix
## Prioritized Task List
## Deferred and Out-of-Scope Items
## Source Notes
```

Under `## Concern Trace Matrix`, create a markdown table with exactly these columns:

```markdown
| Concern | Evidence Path | Requirement ID | Subsystem | Severity | v0.3 Dependency | Recommended Disposition |
```

Include rows for high-priority concerns and test gaps from `.planning/codebase/CONCERNS.md`, especially:
- configuration load process termination
- shared temp resource extraction
- configuration reload and singleton/thread-local ephemeris interactions
- script trusted-code documentation and interruption behavior
- live render-path/layering test gap
- GLB material/lighting/fallback verification gap
- wide-cone star tile lookup gap
- Gaia missing source index and cache behavior
- model conversion fixture-test gap
- deterministic replay/render manifest/performance telemetry foundations

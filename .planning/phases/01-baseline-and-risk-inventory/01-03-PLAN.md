# Plan 03: Map Touchpoints and Phase Boundaries

<objective>
Augment the Phase 1 inventory with package-level touchpoints, representative files, confidence labels, candidate seams, and explicit do-not-touch-yet boundaries.
</objective>

<threat_model>
This plan only edits the Phase 1 inventory artifact. Risks are over-prescribing implementation, encouraging premature refactors, and omitting boundaries that prevent Phase 1 scope creep.

Mitigations:
- Use package-level areas plus representative files, not exhaustive exact fix lists.
- Add confidence labels based on codebase-map evidence.
- Identify candidate seams without asking execution to refactor them.
- Explicitly list deferred modules/features that should not be changed in Phase 1.
</threat_model>

<must_haves>
<truths>
- D-13: Touchpoints should be package-level areas plus representative files.
- D-14: Every touchpoint should include confidence: `High`, `Medium`, or `Low`.
- D-15: Phase 1 should list do-not-touch-yet boundaries for deferred risky modules/features and out-of-scope v0.3 features.
- D-16: Large classes get candidate seams only; no refactors in Phase 1.
</truths>
<outputs>
- `.planning/phases/01-baseline-and-risk-inventory/01-INVENTORY.md` includes touchpoint and boundary sections.
</outputs>
</must_haves>
